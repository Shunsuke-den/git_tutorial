Git Tutorial

Gitチュートリアル

Gitの基本コマンドとワークフローを、ハンズオン形式で学べます。

  • よく使うコマンド
  • コラボレーション
  • 実践ワークフロー
1

よく使うGitコマンド

レポジトリの状態確認・差分確認など、日常的によく使うコマンドです。

🔍

現在の状態を確認する

変更されたファイルやコミットされていないファイルを確認します。

git status
📄

差分を確認する

どの行が変更されたか、コミット前に確認します。

git diff
✏️

直前のコミットを修正する

コミットメッセージの修正やファイルの追加漏れに使います。

git commit --amend
2

コラボレーションのための操作

GitHubを使った開発でよく出てくるキーワードです。

⬇️

リモートの変更を取り込む

GitHub上の最新の変更をローカルに取り込みます。

git pull
🔀

ブランチをマージする

別ブランチで行った作業を現在のブランチに統合します。

git merge
⚡

コンフリクト（競合）

同じ場所を複数人が編集したときに発生する「競合」のことです。マージ時に手動で解決します。

🚀

Pull Request と GitHub Flow

ブランチで作業 → PR作成 → レビュー → マージ、という一連の流れを「GitHub Flow」と呼びます。

3

実践ワークフロー（GitHubにプッシュまで）

チャットで実際に実行したコマンドを、手順の順番で確認できます。

初めてのリポジトリや「fatal: not a git repository」が出たときは、ステップ 1 から。すでに Git が使える状態なら ステップ 3 から始めましょう。

Step 1

現在の状態を確認する

まず、今いるフォルダが Git 管理されているか、どんな変更があるかを確認します。

git status

「fatal: not a git repository」と出た場合は、このフォルダにはまだ Git が入っていません。次のステップへ進みましょう。

Step 2

リポジトリを用意する（必要なときだけ）

初めてこのフォルダで Git を使うときや、.git が壊れているときは、新しくリポジトリを作ります。

git init

「Initialized empty Git repository in ...」と出ればOK。以降、git status が使えるようになります。

Step 3

変更をステージングする

コミットに含めたいファイルを「ステージング」します。全部まとめて追加する場合:

git add -A

特定のファイルだけ追加する場合:

git add index.html

git status で「Changes to be committed」にファイルが出ていれば成功です。

Step 4

コミットする

ステージングした内容を、ひとまとまりの履歴として保存します。-m のあとにメッセージを書きます。

git commit -m "最初のホームページを作成"

「1 file changed」のように表示されれば、コミット完了です。

Step 5

ブランチ名を確認する（プッシュ前に）

GitHub にプッシュするとき、通常は main または master ブランチを使います。どちらか確認します。

git branch

* main なら main、* master なら master にプッシュします。

Step 6

リモート（GitHub）を登録する

GitHub でリポジトリを作成したあと、その URL を「origin」という名前で登録します。

git remote add origin https://github.com/あなたのユーザー名/git_tutorial.git

※「あなたのユーザー名」と「git_tutorial」は自分のものに置き換えます。すでに登録済みの場合は不要です。

Step 7

GitHub にプッシュする

ローカルのコミットを GitHub に送ります。-u origin main で「今後は origin の main にプッシュする」と覚えさせます。

git push -u origin main

ブランチが master の場合は git push -u origin master にします。

この流れをマスターしたら... ファイルの更新 → git add -A → git commit -m "メッセージ" → git push の繰り返しで、いつでも GitHub を更新できます。
